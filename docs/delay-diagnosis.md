# 飞书消息推送延迟诊断报告

**诊断时间：** 2026-03-08 18:21  
**问题：** 飞书消息推送有明显延迟

---

## 时间线分析

### 案例：2026-03-08 10:25 消息

| 时间点 | 事件 | 延迟 |
|--------|------|------|
| 10:25:25.744 | 消息接收 | - |
| 10:25:25.751 | 分发到 Agent | 7ms ✅ |
| 10:26:24.044 | Agent 开始执行 | **58 秒** ❌ |
| 10:26:28.139 | 开始推送 | 4 秒 |
| **总计** | | **~62 秒** |

---

## 根本原因

### 1. 队列阻塞（主要原因）

**证据：**
```
embedded run prompt end: ... durationMs=182340
```

**分析：**
- 前一个任务执行了 **182 秒**（3 分钟）
- 新消息进入队列等待
- 等待时间 = 前序任务剩余执行时间

**影响：**
- 复杂任务（多工具调用、文件操作）会阻塞后续消息
- 队列中只能串行处理

### 2. 模型 API 响应时间

**当前模型：** qwen3.5-plus

**典型响应时间：**
- 简单问题：2-5 秒
- 复杂问题（多工具）：10-60 秒
- 文件操作：5-30 秒/次

### 3. 飞书 API 限制

**当前配置：** WebSocket 长连接

**限制：**
- 推送速率限制：约 10 条/秒
- 单条消息大小限制
- 卡片消息需要额外渲染时间

---

## 责任归属

| 环节 | 延迟占比 | 是否可优化 |
|------|---------|-----------|
| **OpenClaw 队列** | 80%+ | ✅ 是（主要瓶颈） |
| 模型 API | 15% | ⚠️ 部分（选更快的模型） |
| 飞书推送 | 5% | ❌ 否（已是最优） |

---

## 解决方案

### 方案 1：增加并发（推荐）

**修改配置：**
```json
{
  "agents": {
    "defaults": {
      "maxConcurrent": 8  // 当前值：4
    }
  }
}
```

**效果：** 允许同时处理 8 个任务，减少队列等待

### 方案 2：任务优先级

**实现：**
- 简单消息（聊天）优先处理
- 复杂任务（文件操作）后台执行
- 进度定期推送

### 方案 3：模型选择优化

**策略：**
- 简单问题 → 快速模型（qwen3.5-plus）
- 复杂分析 → 强模型（qwen3-max）
- 代码 → 专用模型（qwen3-coder）

配合 Model Router Skill 自动选择

### 方案 4：进度反馈优化

**实现：**
- 长任务开始时立即回复"正在处理..."
- 每 30 秒推送进度更新
- 完成时推送最终结果
